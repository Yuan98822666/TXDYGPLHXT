# 一、整体架构思想

这是一个 **长期运行的市场监控系统**，所以后端必须分层：

```
接口采集层
↓
原始事件层（Raw Event）
↓
状态计算层（State）
↓
信号层（Signal）
↓
API 层（给前端）
```

而且要满足：

- **采集频率可配置**
- **计算频率可配置**
- **前端控制开关**
- **未来可并行 / 可扩展**

# 三、为什么这样分？

## 1️⃣ models 只描述“世界是什么样”

- 不写业务逻辑
- 不写判断
- **只描述数据结构**

👉 这就是你要的“事后重算世界”。

------

## 2️⃣ collectors 永远不直接写数据库

- collector 只负责：
  - 请求接口
  - 解析字段
- **写库统一走 service**

这是为了：

- 防止采集逻辑和存储逻辑耦合
- 以后方便加 Kafka / 消息队列

------

## 3️⃣ scheduler 是“心跳”

- 控制：
  - 1 分钟拉 1 次
  - 1 分钟拉 5 次
- 前端只改配置，不碰代码

------

## 4️⃣ API 层是“遥控器”

- 前端只通过 API 控制系统
- 不允许前端碰采集逻辑

------
